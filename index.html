Containers vs Virtual Machines

What is a Container?

A container is a standardized unit of software that packages code and all its dependencies so the application runs quickly and reliably from one computing environment to another.

Key Characteristics:

  • Isolation: Process-level isolation using kernel features
  • Portability: Runs consistently across different environments
  • Lightweight: Shares host OS kernel, minimal overhead
  • Fast Startup: Starts in seconds, not minutes
  • Microservices: Ideal for microservices architecture

What is a Virtual Machine (VM)?

A virtual machine is a software emulation of a physical computer that runs an operating system and applications. Each VM includes a full copy of an operating system, applications, and virtualized hardware.

Key Characteristics:

  • Full Isolation: Complete OS-level isolation
  • Hardware Virtualization: Emulates physical hardware
  • Heavyweight: Significant resource overhead
  • Slower Startup: Boots entire OS, takes minutes
  • Monolithic: Better for traditional applications

Why Docker? What Problem Does It Solve?

Docker solves the "it works on my machine" problem by ensuring applications run identically across all environments through containerization.

Problems Solved by Docker:

  • Environment Consistency: Eliminates "works on my machine" issues
  • Dependency Management: Packages all dependencies together
  • Deployment Speed: Rapid deployment and scaling
  • Resource Efficiency: Better utilization of system resources
  • DevOps Enablement: Facilitates CI/CD pipelines
  • Microservices Support: Ideal architecture for modern apps

Before Docker (Problems):

  • Different environments (dev, test, prod) with inconsistencies
  • Dependency conflicts and version issues
  • Long setup and configuration times
  • Difficult to replicate issues across environments
  • Resource-intensive virtualization

With Docker (Solutions):

  • Identical environments across all stages
  • Isolated dependencies per application
  • Quick setup with container images
  • Easy replication and debugging
  • Efficient resource usage

Docker Architecture

Docker Architecture - Explanation of Different Layers

Docker uses a client-server architecture with several components working together to build, run, and manage containers.

1. Docker Client

The primary way users interact with Docker. The client sends commands to the Docker daemon, which carries them out.

  • Command-line interface (CLI)
  • REST API for programmatic access
  • Can connect to local or remote daemons

2. Docker Daemon (dockerd)

The core Docker engine that runs on the host machine and manages Docker objects like images, containers, networks, and volumes.

  • Listens for Docker API requests
  • Manages Docker objects
  • Can communicate with other daemons

3. Docker Registry

A storage and distribution system for Docker images. Docker Hub is the default public registry, but you can run your own private registry.

  • Stores Docker images
  • Public (Docker Hub) and private options
  • Image versioning and distribution

4. Docker Objects

The various components that Docker manages, including images, containers, networks, volumes, and plugins.

  • Images: Read-only templates used to create containers
  • Containers: Runnable instances of images
  • Networks: Communication channels between containers
  • Volumes: Persistent data storage

How Does Docker Engine Work in Detail?

Docker Engine is the core of the Docker system, responsible for creating and running containers. It consists of three major components:

Docker Daemon

A persistent background process that manages Docker objects and handles container operations.

containerd

A core container runtime that handles the lifecycle of containers (start, stop, pause, etc.).

runc

A lightweight CLI tool for spawning and running containers according to the OCI specification.

Key Technologies Used by Docker Engine:

Namespaces

Provide isolation for running processes (PID, network, mount, etc.)

Control Groups (cgroups)

Limit and account for resource usage (CPU, memory, I/O)

Union File Systems

Create lightweight images using layers (OverlayFS, AUFS)

Container Format

Combines namespaces, cgroups, and rootfs into a container

Steps of Docker Engine Operation

1

Command Reception

The Docker client sends a command (e.g., docker run) to the Docker daemon via REST API.

2

Image Resolution

Docker checks if the required image exists locally. If not, it pulls the image from a registry (like Docker Hub).

3

Container Creation

Docker creates a writeable container layer over the image's read-only layers using a union filesystem.

4

Network Setup

Docker creates a network interface and connects the container to the specified network (bridge by default).

5

Resource Allocation

Docker sets up control groups (cgroups) to manage and limit the container's resource usage.

6

Process Execution

Docker executes the specified command in the container using namespaces for isolation.

7

Management & Monitoring

The Docker daemon continuously monitors the container and manages its lifecycle.

Example: docker run nginx Flow

$ docker run nginx
# 1. Client sends command to daemon
# 2. Daemon checks for nginx image locally
# 3. If not found, pulls from Docker Hub
# 4. Creates a new container from the image
# 5. Allocates a filesystem and mounts read-only layer
# 6. Creates network interface and connects to bridge
# 7. Starts the container and runs /docker-entrypoint.sh
# 8. NGINX process starts in the container
